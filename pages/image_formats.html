Lesson: Image Formats for Use in HTML

Image File Types / Brief History

Study this lesson and then take the quiz: Image Formats for Use in HTML.

BMP (Bitmap)

  • Creation: Developed by Microsoft in the 1987 for Windows 2.0 
  • Characteristics: Stores images in a raw, uncompressed, lossless format, resulting in high quality but very large file sizes. 
  • Use: Historically used as a default for Windows, it's now less common due to its inefficiency for web use.

BMP is short for bitmap, because that's literally what it is: a raw map of all the pixels in the image. It has no compression. It is is lossless which means it preserves the image 100%. This also means that the files sizes are huge, which made it impractical for use on the Internet.

ICO (Icon)

  • Creation: Developed by Microsoft in the 1987 for Windows 2.0 
  • ICO has exactly the same format as BMP. Icon files are typically 16×16, 32×32, 48×48, or 256×256 pixels. You convert a BMP file to ICO simply by resizing it and renaming it to have the ico extension instead of bmp. You can then use the icon for what is known as a favicon. That is the icon displayed in the browser title bar of your website by putting a link like this in the head section: <link rel="icon" type="image/x-icon" href="/images/favicon.ico">. Read the details here: https://www.w3schools.com/html/html_favicon.asp

GIF (Graphics Interchange Format)

  • Creation: Invented by Steve Wilhite in 1987.
  • Best for: Simple animations, images with a limited color palette (up to 256 colors), and small icons.
  • Characteristics: Supports animation and basic transparency. Its limited color palette makes it less suitable for photographs.

GIFs are still very popular for simple graphics and animations. Animated photos can look pixelated when you zoom in. While they support transparency, they don't support transparency in animation. They have large file sizes compared to the content they deliver, so are not efficient. Modern alternatives that support transparency and animation include APNG (multi-image PNG), WebP, and AVIF.

JPEG / JPG (Joint Photographic Experts Group)

  • Creation: The standard was introduced in 1992.
  • Best for: Photographs and images with continuous tones and complex color variations. The most widely used image format for digital photography and web use.
  • Characteristics: Uses lossy compression, which means some data is discarded during compression to achieve smaller file sizes. This can lead to a slight loss of quality, but it's generally unnoticeable in photographs. Does not support transparency.

PNG (Portable Network Graphics):

  • Creation: PNG was created in 1995 by the PNG Development Group as a replacement / improvement over GIF for static images: better compression, full color, transparency, no patent encumbrances.
  • PNG did not originally support animation (though later APNG etc fixed that).
  • Best for: Images requiring transparency (e.g., logos, icons, graphics with sharp edges), and images with fewer colors.
  • Characteristics: Uses lossless compression, preserving all image data and ensuring high quality. Supports alpha transparency, allowing for varying levels of opacity. File sizes can be larger than JPEGs for complex images.

APNG is the same format as PNG and the same file extension, "png", but it contains multiple PNG images for animation. A browser which does not understand APNG would simply show the first PNG frame as a static image.

SVG (Scalable Vector Graphics):

  • Creation: World Wide Web Consortium (W3C) first created in 2001
  • Best for: Diagrams, icons, logos, and illustrations that need to scale without losing quality.
  • Characteristics: A vector format, meaning images are defined by mathematical equations rather than pixels. This allows for infinite scalability without pixelation. Can be embedded directly into HTML or used as an external file.

WebP (Web Picture format):

  • Creation: WebP (from Google), introduced around 2010, intended to combine advantages of JPEG, PNG, GIF (lossy, lossless, transparency, animation) with better compression.
  • Best for: Both still images and animated images, offering superior compression compared to JPEG and PNG while maintaining quality.
  • Characteristics: Supports both lossy and lossless compression, as well as transparency and animation. Offers smaller file sizes and faster loading times.

AVIF (AV1 Image File Format):

  • Creation: Alliance for Open Media (AOMedia) finalized the first version in 2019. 
  • Best for: High-performance images and animated images, providing excellent compression and quality.
  • Characteristics: A newer format offering even better compression efficiency than WebP. Supports transparency and animation. It uses the AV1 codec (for high compression efficiency) stored in a HEIF container. It supports both lossy & lossless, good color depths, etc.

Choosing the Right Format:

  • Photographs: JPEG or WebP (for better compression).
  • Logos, Icons, Graphics with Transparency: PNG, SVG, or WebP.
  • Animations: GIF (for simple animations), WebP, or AVIF.
  • Scalable Graphics: SVG.

JPG, GIF, and PNG are the still the most common formats on the Internet, however modern formats WebP and AVIF with superior compression, quality, and features are growing in popularity.

Key Concepts and Technical Terms

  • Raster vs vector: Raster images are pixel-based (like JPG, PNG), vector are geometry/path based (SVG). Raster are limited by resolution; vector scale nicely.

  • Lossy vs lossless compression: Lossy = some data / detail discarded to reduce file size (JPEG lossy, WebP lossy, AVIF lossy versions). Lossless = image is represented without losing information (PNG, lossless WebP, lossless AVIF).

  • Color depth: Number of bits per pixel / per channel (8-bit per channel, 10/12/16 bits, etc), how many colors can be represented, also whether color space / gamut / HDR etc are supported.

  • Transparency / Alpha channel: Whether pixels can be partially transparent (e.g. PNG, WebP, AVIF) vs only fully on/off or none.

An alpha channel is an additional data layer in digital images and videos that provides transparency information, controlling how opaque or transparent each pixel is. Represented as a grayscale mask, it determines whether parts of the image are fully visible, fully transparent (invisible), or partially see-through, allowing for effects like compositing, layering, and special effects by blending with other images or backgrounds.

  • Animation: Some formats support multiple frames. GIF has long supported, APNG adds that for PNG‚Äêbased images, WebP/AVIF have animation support in many cases.

  • Compression artifacts: Lossy compression can introduce blur, blockiness, color banding, especially with sharp transitions.

  • Progressive rendering / interlacing: Some image formats support progressive loading (e.g. JPEG progressive, interlaced PNG) so that a low resolution or coarse version appears quickly and then refines.

  • Browser support: Not all browsers support all formats or all features of each format (e.g. animation, lossless WebP, AVIF etc).

The <picture> element and Fallback Images

Most images are linked to HTML pages by using the img element, like so:

<img src="img/img_orange_flowers.jpg" alt="flowers"/>

where the attribute src contains the path to the image file and alt contains a text description of the image for accessiblity screen readers.

However, HTML5 introduced the picture element for responsive design, allowing developers to provide different image sources. The picture element is formatted like so, to accommodate screens that have different widths:

<picture>
   <source media="(min-width:650px) srcset="img_ping_flowers.jpg" />
   <source media="(min-width:465px) srcset="img_white_flower.jpg" />
   <img src="img_orange_flowers.jpg" alt="flowers" />
</picture>

If the available width is too small to accommodate a 650px image, it will fallback to display the 465px image instead. If the screen is too small to display even that, it will display the default jpg at the bottom. Try this out yourself at W3Schools: https://www.w3schools.com/tags/tryit.asp?filename=tryhtml5_picture. Resize the window to see the image change.

You could, of course, use styling to resize a picture dynamically to a percentage of the screen, such as 20% of the width. But resizing happens in the client browser after the entire file has been downloaded. Your website will load faster if you don't force the client to waste time downloading images that are larger than it can display, when it could have selected a smaller image instead.

Fallback images can also accommodate browsers that don't support the newest file formats. A developer can support those that do while still providing images for those that don't:

<picture>
   <source srcset="photo.avif" type="image/avif" />
   <source srcset="photo.webp" type="image/webp" />
   <img src="photo.jpg" alt="photo" />
</picture>

A browser that supports the picture element will first attempt to load the first image, which in this case is AVIF, specified by the source element. If it doesn't understand the format, it will drop to the second option, which is WebP. Finally, it will drop to the last option on the list, which is a JPEG, which all browers support. If the browser is very old and doesn't even know what the picture element is, it will ignore the outer tag but still load the JPEG file because that is in an img element. For this reason, an img element with a safe format like JPEG should always be the final fallback option. There should only be one img element in the picture element, because a browser that doesn't understand picture will display all img elements. The alt attribute on the img tag will be used as the alternate text for whichever picture is displayed, which is why there is no alternative text on the source elements.

“Which format is best for what” – Decision Guidelines

Situation / Requirement Best Format Choices Why
Photographs / images with many colors, detailed gradients, scenes (people, landscapes) JPEG (lossy) or next-gen lossy like WebP or AVIF Because lossy compressions work very well for photos, human eye tolerates small imperceptible losses. WebP/AVIF often get same visual quality at smaller size.
Logos, icons, UI graphics, images with sharp lines, text inside the image SVG (if vector) or PNG (lossless) or lossless WebP / AVIF You want crisp edges, exact colors, possibly transparency; lossy JPEG tends to blur edges, introduce artifacts; vector SVG scales without losing quality.
Transparent backgrounds / alpha PNG, WebP, AVIF Need full alpha channel; GIF has only binary transparency; JPEG has none.
Animation / small motion graphic (e.g. a short looping animation) WebP (animated), APNG, GIF (if backward compatibility mandatory) WebP/AVIF animations tend to be more efficient / higher quality; APNG better quality than GIF but size may be larger; GIF is universal but poor quality & palette limitations.
Where file size / bandwidth / page load performance is critical WebP / AVIF (lossy for photos; lossless when required) These modern formats often give best compression per perceptual quality; using multiple formats with fallbacks where needed.
Legacy browser support / maximum compatibility JPEG, PNG, GIF; possibly ICO for icons These are safe bet formats almost universally supported. Use fallback sources.
Scalability (responsive, high DPI / retina) SVG for vector; for raster, generate multiple sizes / responsive formats + modern formats (e.g. WebP/AVIF), include proper srcset / picture To avoid serving an enormous high-res image where not needed; vector avoids scaling issues entirely.

Browser Support Summary

  • PNG, JPEG, GIF, ICO – universally supported in all modern browsers and backwards. No real concern here.

  • SVG – good support across modern browsers; older IE versions had quirks; some mobile-browser-versions might have minor issues especially in scripting/interaction. But generally very safe now.

  • APNG – well supported in current browsers; older ones may fallback to showing only first frame.

  • WebP – broadly supported in recent major browsers; some older or niche browsers or certain mobile OS versions may not support, so fallback strategy often needed.

  • AVIF – support is rapidly increasing. Major browsers support it. But because it’s newer, there may be performance / encoding quality variation; fewer tools historically. Fallbacks are wise if audience may include older browsers.
